## Принципы работы с кодом

### Один модуль — одна ответственность
Не добавляй логику одного слоя в файл другого.
Например: логика эволюции живёт только в `evolution/`, никогда в `core/`.

### Параметры только в конфиге
Никаких магических чисел в коде.
Все пороги, веса, таймауты — в `config/params.yaml`.

```python
# ❌ Плохо
if imbalance > 0.68:

# ✅ Хорошо
if imbalance > self.params.imbalance_threshold:
```

### Комментарии объясняют "зачем"
Код говорит что происходит. Комментарий объясняет почему именно так.

```python
# ❌ Плохо
# делим на сумму объёмов
imbalance = bid_vol / (bid_vol + ask_vol)

# ✅ Хорошо
# Нормализуем к [0,1] чтобы сравнивать между разными уровнями ликвидности
imbalance = bid_vol / (bid_vol + ask_vol)
```

### Типизация строгая, без исключений
- mypy strict mode — никаких `Any`, никаких `# type: ignore` без объяснения
- Все функции с аннотациями аргументов и возвращаемого типа
- Dataclasses или TypedDict для структур данных

```python
# ❌ Плохо
def compute_signal(data):
    return data["imbalance"] > 0.7

# ✅ Хорошо
def compute_signal(data: MarketSnapshot) -> Signal:
    return Signal.LONG if data.imbalance > self.params.imbalance_threshold else Signal.HOLD
```

## Перед тем как писать код

1. Прочитай существующий файл который будешь менять
2. Убедись что понимаешь как он связан с другими модулями
3. Проверь есть ли тест на этот модуль
4. Если задача большая — сначала опиши план, потом пиши

---

## После того как написал код

1. Запусти `ruff check .` — исправь все замечания
2. Запусти `mypy --strict .` — исправь все ошибки типов
3. Запусти тесты `pytest tests/` — убедись ничего не сломал
4. Если добавил новую функцию — добавь тест на неё
